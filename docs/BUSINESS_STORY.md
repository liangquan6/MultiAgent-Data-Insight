# 业务故事: 为什么做这个项目, 解决谁的痛点

> 这是 W5.1 的产出。技术能力本身只回答"你会不会", 业务故事回答"做了有什么用"。
> 简历和面试里, 后者往往更被看重。

## 一句话

> 我做了一个"上传 CSV, 4 个 Agent 自动协作产出分析报告 + 可视化"的系统,
> 拿二手房挂牌数据做演示: 一个不懂 pandas 的中介店长, 把爬来的挂牌表丢进去,
> 能直接拿到"哪个区域性价比高、哪些房源被低估、带看转化怎么看"的报告, 不用养一个数据分析师。

## 解决谁的痛点

### 痛点画像: "有数据, 没分析"的小团队

| 角色 | 现状 | 痛点 |
|---|---|---|
| 中小房产中介店长 | 手上有爬虫/同事整理的挂牌 CSV | 想知道"我这片区哪些房源值得推", 但只会 Excel 透视表, 做不出区域价差/户型性价比这种多维度分析 |
| 教务/运营新人 | 拿到一份学生成绩或课程评分表 | 要交"数据简报", 但不熟 Python, 现学 pandas + matplotlib 来不及 |
| 自媒体/调研作者 | 收集了一堆公开数据 | 想快速出"数据怎么说"的图文, 不想为每篇都手写一套分析代码 |

**共性**: 有数据, 没分析师, 没时间学工具。传统路径是"招个数据实习生"或"自己啃 Python",
成本都高。这个项目想验证的是 —— **能不能用多 Agent 把"会写代码的分析师"这个角色自动化掉一截**,
让非技术用户也能把一张表变成一份能看的报告。

### 这个项目不解决什么(诚实边界)

- 不是 BI 工具替代品: 没有 Tableau/Power BI 那种拖拽交互, 产出是静态报告 + 图。
- 不接实时数据源: 现在是上传 CSV, 不是连链家/贝壳 API 实时拉盘。
- 完成率有上限: W4 实测 4 数据集 × 3 模式 12 次跑批, 只有 25% 跑出完整报告
  (受消息上限 + 返工循环制约, 见 README 对比表)。这是 Agent 协作的真实瓶颈,
  也是这个项目诚实的"当前能力天花板", 不是 demo 失败。

## 演示场景: 长沙二手房挂牌分析

### 数据 (data/changsha_housing.csv)

300 条合成挂牌, 13 字段, 贴近真实结构:

- **分类**: 区域(7 区)、户型(6 类)、朝向(6 方位)、装修(精装/简装/毛坯)
- **数值**: 面积、总价(万)、单价(元/㎡)、楼层、总楼层、建成年代、关注人数、带看次数
- **脏数据**: 朝向缺 6%、装修缺 5%、带看次数缺 11% (模拟挂牌太久没人看的房源)
- **隐含派生**: 房龄 = 当前年 - 建成年代; 单价应 ≈ 总价×10000/面积 (Agent 需自行核验)

价格生成逻辑贴近真实: 区域档位(芙蓉 13500 > 长沙县 7800) × 面积(大户型单价略低)
× 房龄(每年 -1.2%) × 装修(精装 +5%) + 个盘噪声。实测区域均价梯度正确(芙蓉 12123 > 长沙县 6712)。

### 一个使用故事(90 秒能讲完)

> 长沙某中介店长小张, 手上有 300 条片区挂牌数据, 老板问他"哪些房源性价比高、
> 带看热度怎么看"。小张不会 Python。
>
> 他打开系统, 上传 changsha_housing.csv。4 个 Agent 自动开工:
> - **Analyst** 读表头, 判断"单价有区域差异、带看次数有缺失, 建议按区域分组看性价比";
> - **Coder** 调可视化工具, 出"区域均价箱线图""面积-单价散点图""带看次数分布";
> - **Reviewer** 检查图, 发现 Coder 调 detect_outliers 没传 column 参数, 当场打回重做;
> - **Reporter** 汇总成 markdown 报告。

### 实跑结果 (round_robin, 真实数据)

实际跑了一次 (results/changsha_housing_report.md), 耗时 136.5s, 46 轮, 148k token,
生成 29 张图。Reporter 产出的报告结构:
- **区域格局**: 岳麓区单价最高(¥11,820/㎡)且波动最大, 开福区是洼地(¥8,230/㎡)
- **房龄效应**: 越新越贵, 2015 后新房均 ¥12,150 vs 2000 前老房 ¥7,420, 价差 64%
- **户型策略**: 3室2厅是成交主力(总价中位 ¥138 万), 4室改善溢价 56%
- **流量转化**: 关注人数 vs 带看次数 皮尔逊 r=0.83, 强正相关
- **装修溢价**: ANOVA 显示精装比毛坯贵 42%, 简装比毛坯贵 14.8%

报告带具体数值、ANOVA/Tukey 统计检验、9 张图证据路径链接, 可追溯可复验。
小张拿到这份报告 + 29 张图, 直接贴进周会 PPT。全程没写一行代码。

> **诚实注**: 这次跑批触发了"团队协作未产出报告"兜底 (46 轮吃满消息上限后,
> 单独调 Reporter 重写)。这印证了 W4 的发现 —— 返工循环会吃掉收尾窗口。
> 但兜底机制保证了用户始终能拿到一份报告, 不至于白跑。这也是工程上"保可用"的一层。

### 这个故事对面试官的价值

1. **不是又一个 RAG/聊天 demo**: 多 Agent 协作 + 工具调用 + 评估闭环, 技术栈完整。
2. **有真实业务锚点**: 二手房分析是听得懂的场景, 不用解释"为什么需要 4 个 Agent"。
3. **有诚实的负面数据**: 完成率 25%、LLM 自调度 75% 循环率, 说明做过实测、知道边界在哪,
   不是只报喜的 toy project。面试官更信"知道哪里不行"的人。

## 为什么是"多 Agent"而不是单 Agent

单 Agent (一个 ReAct loop) 也能调工具出图, 为什么拆 4 个? 这个项目的回答:

1. **职责分离降低单 prompt 复杂度**: Analyst 只管"看什么", Coder 只管"怎么画",
   Reviewer 只管"对不对", Reporter 只管"怎么讲"。每个 Agent 的 system message 短而专,
   比塞进一个超长 prompt 的单 Agent 更稳定 (W2 实测单 Agent 易跑偏到无关分析)。
2. **返工可控**: Reviewer 发现问题能让 Coder 重画, 是显式的协作回路;
   单 Agent 自己画自己评, 没有外部校验, 错了容易一路错到底。
3. **可评估**: 4 个角色让"完成率""一次通过率""工具误用"有明确观测点
   (Reporter 是否产出 = 完成; Reviewer 是否返工 = 一次通过; Coder 工具调用 = 误用)。
   单 Agent 这些指标很难定义。这也是 W4 benchmark 能成立的前提。

> 代价也诚实记: 多 Agent 协作有调度开销, W4 实测 RoundRobin 平均 38 轮 / 271k token,
> 比单 Agent 费。这是"稳定性 vs 效率"的取舍, 项目选了前者。

## 和简历能力的对应

| 简历要写的 | 这个故事怎么支撑 |
|---|---|
| 工具编排 | Coder 调 6 个可视化/检测工具, MCP 协议暴露 |
| 记忆管理 | SchemaMemory 压缩表结构, prompt token 降 16.5% |
| Agent 评测 | 4 数据集 × 3 模式 12 次跑批, 6 项指标 |
| 工程取舍 | 排除 MagenticOne (架构错配)、确定性 vs LLM 调度 (实测 75% 循环率) |
| 业务落地 | 二手房场景 + 非技术用户故事, 不止是跑 benchmark |
